# エージェントハーネスの設計パターン

コーディングエージェント（Claude Code, Codex 等）のハーネス設計に関する体系的知識。2 つの補完的フレームワークを統合:

- **12 パターン** (Generative Programmer) — Claude Code リークから抽出した再利用可能な設計パターン。4 カテゴリに分類
- **6 コンポーネント** (Raschka) — コーディングハーネスの構成要素を実装視点で分解。リファレンス実装 [Mini Coding Agent](https://github.com/rasbt/mini-coding-agent) 付き

## 用語体系

Raschka による定義の整理:

| 用語 | 定義 |
|---|---|
| LLM | 生のモデル（次トークン予測） |
| Reasoning Model | 推論時計算を増やした LLM（中間推論・検証・候補探索） |
| Agent | モデル＋ツール＋メモリ＋環境フィードバックのループ |
| Agent Harness | エージェントを囲むソフトウェア足場。コンテキスト・ツール・プロンプト・状態・制御フローを管理 |
| Coding Harness | Agent Harness の特化型。コードコンテキスト・ツール・実行・反復フィードバックに最適化 |

エージェントループの内部構造: **Observe**（環境から情報収集）→ **Inspect**（情報分析）→ **Choose**（次ステップ選択）→ **Act**（実行）

### ハーネスが差別化要因である

Raschka の重要な主張: **2026 年現在、最先端 LLM のバニラ性能は収束しつつあり（GPT-5.4, Opus 4.6, GLM-5）、ハーネスこそが実用上の差別化要因**。同じモデルでもチャット UI とコーディングハーネスでは体感性能が大きく異なる。「モデル品質」と見なされているものの多くは実際には「コンテキスト品質」。

## Memory & Context（記憶とコンテキスト）

### 1. Live Repo Context（ライブリポジトリコンテキスト）

Raschka の 6 コンポーネントの第 1。セッション開始時にエージェントが自動収集する「安定した事実」のワークスペースサマリー: Git リポジトリかどうか、ブランチ、ステータス、プロジェクトドキュメント（README, CLAUDE.md 等）、ディレクトリ構造。「fix the tests」のような指示を自己完結的に解釈するための前提情報。

### 2. Project Instructions File（プロジェクト指示ファイル）

リポジトリに同梱される永続的な設定ファイル（例: `CLAUDE.md`, `AGENTS.md`）。ビルドコマンド、テストコマンド、アーキテクチャルール、命名規約、コーディング標準を定義し、セッション開始時に自動ロードする。

- **解決する問題**: セッションごとに同じ指示を繰り返す必要がある
- **トレードオフ**: プロジェクト進化に合わせたメンテナンスが必要。陳腐化した指示ファイルは指示なしより悪い

### 3. Hierarchical Scoped Instructions（階層スコープ指示）

組織・ユーザー・プロジェクトルート・親ディレクトリ・子ディレクトリなど複数スコープから動的に指示をロードする。import 構文で分割も可能。

- **解決する問題**: 単一指示ファイルがモノレポや多言語プロジェクトで巨大化 or 汎用化
- **トレードオフ**: 発見可能性の低下。スコープ間のルール衝突で予期しない動作が起こりうる

### 4. Prompt Shape & Cache Reuse（プロンプト構造とキャッシュ再利用）

Raschka の 6 コンポーネントの第 2。プロンプトを「安定プレフィックス」と「変動部分」に分離:

| 部分 | 内容 | 更新頻度 |
|---|---|---|
| 安定プレフィックス | 一般指示、ツール記述、ワークスペースサマリー | ほぼ不変 |
| 変動部分 | 短期メモリ、直近トランスクリプト、最新ユーザーリクエスト | 毎ターン |

安定プレフィックスをキャッシュし再利用することで、毎ターンの再処理コストを削減する。「スマートなランタイムは毎ターン巨大な未分化プロンプトを再構築しない」(Raschka)。

### 5. Tiered Memory（階層化メモリ）

メモリを 3 層に分離:

| 層 | ロード戦略 | 例 |
|---|---|---|
| コンパクトインデックス | 常時コンテキスト内（200行上限） | `MEMORY.md` |
| トピック別ファイル | オンデマンド（タスク関連時） | `feedback_testing.md` |
| セッション全文 | ディスク上、必要時のみ検索 | セッション履歴 |

Raschka はセッション状態を 2 層に整理: **working memory**（小さく蒸留された状態。タスク・重要ファイル・最近のメモ）と **full transcript**（全リクエスト・ツール出力・LLM応答の完全記録、JSON ファイルでディスク保存）。compact transcript（プロンプト再構築用）と working memory（タスク継続用）は役割が異なる。

- **解決する問題**: 全メモリのコンテキストロードはトークン浪費、サイズ制限超過、ノイズ埋没を引き起こす
- **トレードオフ**: 何をどの層に置くか、層間の昇格・降格のタイミング、インデックス同期の複雑さ
- **関連**: [LLM Wiki パターン](llm-wiki-pattern.md)も知識の永続化と構造化に取り組むが、対象は外部知識。こちらはエージェント自身の動作記憶

### 6. Background Memory Consolidation（バックグラウンドメモリ整理）

アイドル時にバックグラウンドプロセスがメモリを重複排除・剪定・再構成する。Claude Code の「autoDream」モードでは 8 フェーズのメモリ管理と 5 種のコンテキスト圧縮が確認された。

- **解決する問題**: 重複蓄積、矛盾する古い事実、インデックス肥大
- **トレードオフ**: 整理プロセス自体のトークン消費と誤判断リスク。過剰な剪定はユーザーが必要とする情報を削除しうる

### 7. Progressive Context Compaction（段階的コンテキスト圧縮）

会話の古さに応じて複数段階の圧縮を適用:

1. **最近のターン** — フル詳細を保持
2. **やや古いターン** — 軽い要約
3. **非常に古いターン** — 積極的な圧縮

Claude Code では HISTORY_SNIP → Microcompact → CONTEXT_COLLAPSE → Autocompact の 4 層が使われている。

Raschka は最小限のハーネスでも少なくとも 2 つの圧縮戦略が必要だと指摘: **クリッピング**（長いドキュメント断片・ツール出力・メモリノートの切り詰め）と **トランスクリプト削減**（セッション履歴の要約）。加えて、古いファイル読み取りの**重複排除**で同一ファイルの繰り返し表示を防ぐ。

- **解決する問題**: 長セッション（20-30 ターン超）でのコンテキストウィンドウ枯渇。コーディングエージェントは繰り返しのファイル読み取り・長いツール出力・ログにより、通常のチャットよりコンテキスト膨張しやすい
- **トレードオフ**: 非可逆圧縮。要約ステップのたびに詳細が失われ、圧縮済みセグメントの情報が必要になった場合、ハルシネーションの原因になる

## Workflow & Orchestration（ワークフローとオーケストレーション）

### 8. Explore → Plan → Act（探索 → 計画 → 実行）

ワークフローを 3 フェーズに分離し、書き込み権限を段階的に拡大:

| フェーズ | 許可される操作 |
|---|---|
| Explore | 読み取り、検索、コードベースのマッピングのみ |
| Plan | ユーザーとのアプローチ議論 |
| Act | フルツールアクセス（編集・実行） |

- **解決する問題**: 不完全な理解のまま編集に入ることで、誤ファイルの変更・依存関係の見落とし・既存パターンの無視が起こる
- **トレードオフ**: 単純なタスクに対して探索・計画フェーズが冗長に感じられる

### 9. Context-Isolated Subagents（コンテキスト分離サブエージェント）

リサーチ・計画・編集を別々のエージェント（独自のコンテキストウィンドウ・システムプロンプト・制限ツールアクセス）で実行する。リサーチエージェントはコード編集不可、計画エージェントはコマンド実行不可。

Raschka はサブエージェント設計の核心的問題を「十分なコンテキストを継承させつつ、どう制約するか」と定式化。サブエージェントは親のコンテキストを継承するが、より狭い境界（リードオンリー、再帰深度制限等）で動作する。Codex はリードオンリー強制ではなく、タスクスコープ・コンテキスト・深度で境界を設ける方式。

- **解決する問題**: 長セッションで研究結果・計画議論・コード編集・テスト出力が混在し、コンテキストが汚染される
- **トレードオフ**: 調整オーバーヘッド。サブエージェントへの情報受け渡しで重要なニュアンスが失われうる
- **関連**: [Agentic RAG](agentic-rag.md) の Multi-Agent Router パターンも複数エージェント間の調整に取り組む

### 10. Parallel Worktree Execution（並列ワークツリー実行）

独立したサブタスクを並列で実行。各サブエージェントは独立した git worktree（リポジトリの隔離コピー）で作業する。親のキャッシュコンテキストを再利用するため、並列分岐のトークンコストは実質ゼロ。

- **解決する問題**: 依存関係のない 20 ファイルの移行が 20 逐次ステップになる
- **トレードオフ**: 並列ブランチが重複ファイルに触れた場合、マージコンフリクトが逐次作業より困難になる

## Tools & Permissions（ツールと権限）

### 11. Progressive Tool Disclosure（段階的ツール開示）

デフォルトで少数のツールセット（Claude Code では 20 未満）を提供し、必要に応じて追加ツールをアクティベートする。デフォルトは Read, Edit, Write, Bash, Grep, Glob 等。MCP ツール・リモートツール・カスタムスキルは必要時のみ。

- **解決する問題**: 60 ツール全表示ではモデルの選択コストが増大し、誤ツール選択の確率が上がる
- **トレードオフ**: 拡張ロジックの複雑さ。遅すぎるアクティベーションはエージェントが適切なツールなしで無駄なターンを費やす

### 12. Deterministic Permission Gating（決定論的権限ゲーティング）

実行前にツールごとの allow / ask / deny ルールとパターンマッチングで決定論的に権限を判定する。シェルコマンドは動詞・フラグ・対象を解析してリスク分類する。

Raschka のツールフローの全体像: モデルが構造化アクションを出力 → ハーネスが検証（既知ツールか？引数は有効か？ワークスペース内のパスか？）→ 承認が必要なら要求 → 実行 → 結果を境界内でループに返す。「ハーネスがモデルに与える自由度を減らすことが、同時に利便性を向上させる」(Raschka)。

- **解決する問題**: 全コマンド承認は危険。全コマンド確認はファティーグ（ユーザーが全て「yes」を押すようになる）
- **トレードオフ**: 決定論的分類器は全ての安全/危険コマンドを予測できない。継続的なルール調整が必要

### 13. Structured Tool Abstraction（構造化ツール抽象化）

汎用シェル（cat, sed, grep, find）の代わりに、目的別の専用ツール（FileReadTool, FileEditTool, GrepTool, GlobTool）を提供する。各ツールは型付き入力・制約されたスコープ・独自の権限ルールを持つ。Raschka はこれを「事前定義ツールと検証済み入力による明確な境界」と表現。ただし `subprocess.call` 等による汎用シェルもフォールバックとして残す。

- **解決する問題**: 汎用シェル経由のファイル操作はレビュー困難、権限設定困難、モデルが正しく使いにくい
- **トレードオフ**: 専用ツールは全エッジケースをカバーできない。汎用シェルをフォールバックとして残す必要がある

## Automation（自動化）

### 14. Lifecycle Hooks（ライフサイクルフック）

エージェントのライフサイクルの特定ポイントでシェルコマンドやアクションを自動実行する。プロンプト指示ではなくフックとして実装することで、確実に毎回実行される。

Claude Code では 25 以上のフックポイント: PreToolUse, PostToolUse, SessionStart, CwdChanged 等。

- **解決する問題**: フォーマッタ実行・コマンド検証・設定リロードなど「毎回必ず」の動作をプロンプト指示に頼ると、モデルが忘れる・スキップする・再解釈する
- **トレードオフ**: デバッグ困難。フック内の問題は会話外で発生するため、プロンプトレベルの指示より診断しにくい

## 2 フレームワークの対応関係

Raschka の 6 コンポーネントと 12 パターンの対応:

| Raschka の 6 コンポーネント | 対応するパターン |
|---|---|
| 1. Live Repo Context | #1 Live Repo Context, #2 Project Instructions File, #3 Hierarchical Scoped Instructions |
| 2. Prompt Shape & Cache Reuse | #4 Prompt Shape & Cache Reuse |
| 3. Structured Tools, Validation, Permissions | #11 Progressive Tool Disclosure, #12 Deterministic Permission Gating, #13 Structured Tool Abstraction |
| 4. Context Reduction & Output Management | #7 Progressive Context Compaction |
| 5. Transcripts, Memory, Resumption | #5 Tiered Memory, #6 Background Memory Consolidation |
| 6. Delegation & Bounded Subagents | #9 Context-Isolated Subagents, #10 Parallel Worktree Execution |

12 パターンにあって Raschka が明示的にカバーしていないもの: #8 Explore→Plan→Act、#14 Lifecycle Hooks。

## パターン間の関係

14 項目は独立ではなく相互補完的:

- **Memory & Context の 7 項目**は「リポジトリ事実収集 → 静的指示ファイル → スコープ分離 → プロンプトキャッシュ → メモリ階層化 → バックグラウンド整理 → コンテキスト圧縮」という進化を形成
- **Workflow パターン**は読み書きの分離（Explore→Plan→Act）、コンテキストの分離（Subagents）、作業の並列化（Worktree）の 3 軸
- **Tools & Permissions**はツール数の制御（Progressive Disclosure）、実行権限の制御（Permission Gating）、インターフェースの制御（Structured Abstraction）の 3 層
- **Lifecycle Hooks**はカテゴリ横断的。メモリ・ワークフロー・ツール全てに適用される自動化基盤
- Raschka が強調するように、6 コンポーネントは実装上「高度に絡み合って」おり、個別に論じても実装では不可分

## 設計指針

- これらは一時的なトリックや製品固有の機能ではなく、エージェントハーネス設計の**基盤的アーキテクチャ決定**
- メモリ階層化、コンテキスト圧縮、権限ゲーティング、ライフサイクルフックは、モデルやツールが進化しても**パターンとして有効**であり続ける
- エージェントのデフォルト動作は「全てを混ぜる」であり、その混在がタスク規模に応じて品質を劣化させる。パターンの共通テーマは**分離**（関心の分離）
- 「モデル品質」と見なされているものの多くは実際には「コンテキスト品質」(Raschka) — ハーネスの地味な部分（コンテキスト削減・出力管理）が過小評価されがち

## Coding Harness vs General Agent Platform

Raschka は OpenClaw との比較で、コーディングハーネスと汎用エージェントプラットフォームの違いを指摘。両者は指示ファイル・セッション管理・サブエージェント等の共通要素を持つが、**最適化対象が異なる**: コーディングハーネスはリポジトリ内での作業に特化（ファイル検査・コード編集・ローカルツール実行）、汎用プラットフォームは複数ワークスペース・チャンネル横断の長期エージェント実行に最適化。
